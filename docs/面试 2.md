# 项目整体布局与面试备忘

## 一、整体架构速览
- **框架与基础设施**：Hertz 作为 HTTP 框架；GORM/Gen 数据访问；Redis 用于缓存、频控、会话；RabbitMQ 负责异步通知；Snowflake 生成全局 ID；OpenTelemetry/Prometheus/Loki/FuentBit 打通可观测性。
- **进程角色**：
  - `cmd/server`：主 API 服务。
  - `cmd/worker`：消费 MQ 消息（短信/通知等）。
  - `cmd/scheduler`：定时任务调度（打卡、行程提醒）。
  - `cmd/gen`：GORM Gen 代码生成。
- **核心领域**：
  1) **用户/认证**：支付宝手机号授权、验证码登录绑定、waitlist 引导。
  2) **打卡提醒**：每日提醒、通知任务去重、月度频控。
  3) **行程报备**：行程创建、超时提醒延迟消息。
  4) **额度与扣费**：额度钱包、交易流水、新用户默认额度。
  5) **通知链路**：短信/外呼消息的生产与消费。

## 二、代码目录要点
- `internal/service`：业务核心（auth/user/checkin/journey/notification/quota/...）。
- `internal/handler`：HTTP 入口，参数校验、调用 service。
- `internal/middleware`：认证、CSRF、跨域、限流、日志、otel。
- `internal/cache`：Redis 访问封装（用户设置、打卡、验证码、slider、waitlist 缓存与频控）。
- `internal/schedule`：定时调度打卡、行程。
- `internal/queue` & `storage/mq`：MQ 客户端与消费/生产。
- `pkg`：通用组件（logger/metrics/redis/token/sms/slider/oss 等）。
- `schema/schema.sql`：表结构与索引/唯一约束。
- `docker/`：本地与部署镜像配置；`scripts/`：数据库重建、单次调度脚本。

## 三、关键业务流程（摘要）
1) **Waitlist / 基于 open_id 的引导**
   - 入口：`GET /v1/users/waitlist`（`handler/user.go`）。
   - 逻辑：`service/user.go::GetWaitListInfo`  
     - Redis 限流：`waitlist:rate:{open_id}` 60s 内 >30 次拒绝。
     - 缓存：`waitlist:info:{open_id}` 5 分钟。
     - 若不存在用户且容量未满，创建最小用户（状态 `waitlisted`）。
     - 返回统一 `AuthUserSnapshot`，附带 `next_step`：`waitlist|bind_phone|fill_contacts|home`。

2) **支付宝授权换手机号**
   - 入口：`POST /v1/auth/miniapp/alipay/exchange`。
   - 逻辑：`service/auth.go::ExchangeAlipayAuthCode`
     - 解密手机号 → `phone_hash`，优先按 `alipay_open_id` 查用户，找不到再按 `phone_hash` 合并。
     - 新用户：创建用户（状态 `contact`）、初始化额度钱包与交易。
     - 已有用户：回填 `alipay_open_id`、`phone_hash`、状态流转（waitlisted/onboarding → contact）。
     - 返回 token + `next_step`。

3) **验证码注册/登录/绑定**
   - 入口：`/v1/auth/phone/verify`，`/v1/auth/phone/send-captcha`。
   - 逻辑：`service/auth.go::VerifyPhoneCaptchaAndLogin/Bind`
     - 校验验证码 → 查/创用户（同样额度初始化）、状态到 `contact`。
     - 生成 token；返回 `next_step`。

4) **每日打卡提醒**
   - 定时任务：`schedule/checkin_scheduler.go`；业务：`service/checkin.go`
   - 关键点：事务批量创建通知任务；当日去重（按 user/date/category）；月度频控（Redis）；Snowflake 生成任务 ID/code。

5) **行程报备与超时提醒**
   - 入口：`/v1/journeys`（handler/service）。
   - 逻辑：`service/journey.go`
     - 校验返回时间 ≥ 当前+10min。
     - 创建行程；若自动提醒开启且延迟 ≤24h，生成延迟消息（由上层发布）；>24h 交给 scheduler。

6) **额度与扣费**
   - 新用户默认额度：注册事务中创建 `quota_wallets` 与 `quota_transactions`。
   - 唯一约束：`(user_id, channel)`；流水记录扣减/充值，便于对账。

## 四、数据模型与约束（节选）
- `users`：`public_id`、`alipay_open_id`、`phone_hash` 唯一；`status` 索引；`emergency_contacts` GIN。
- `daily_check_ins`：`(user_id, check_in_date)` 唯一。
- `quota_wallets`：`(user_id, channel)` 唯一。
- `notification_tasks`：`task_code` 唯一；多索引支持状态/用户查询。
- `journeys`：`user_id,status` 索引；延迟消息跟踪字段 `timeout_message_id`。

## 五、非功能特性
- 限流：通用中间件（滑动窗口+阻断）；验证码、设置、行程修改等可单独配置。
- 缓存：用户设置、waitlist、验证码/slider；保护层封装空值与防雪崩。
- 观测性：otel 初始化、MQ/DB/Redis otel hook、业务/短信指标；日志统一 zap。
- 部署：Dockerfile 多角色（api/scheduler/worker）；compose 提供 Prometheus/Loki/Otel-collector。

## 六、风险与可优化点（可执行版）
1) Waitlist 缓存失效策略  
   - 在状态流转/绑定手机号/更新 open_id 后，删除 `waitlist:info:{open_id}`；必要时写回最新快照。  
   - 为 `waitlist:rate` 加最大 TTL，防止计数键残留。
2) Redis 故障兜底  
   - 限流/缓存读写异常时，触发本地退化限流或直接返回 429/503，并记录告警。  
   - 将 Redis 错误计数暴露为指标，便于提前发现。
3) 行程并发控制  
   - 在 DB 增加唯一约束/检查：`user_id + status=ongoing`，创建前查询未完成行程直接返回。  
   - 对创建接口加轻量限流（按 user/IP）。
4) 扣费/通知幂等  
   - 消费端以 `notification_tasks.id` 或业务幂等键落地，先查任务状态再扣费/更新，避免重复。  
   - 预扣/确认/退款链路补充一致性校验与重试/死信策略。
5) 输入与日志安全  
   - 对 open_id、昵称等做长度/字符集校验，拒绝超长或异常字符；日志字段做截断/过滤。  
   - 为高频接口增加参数白名单和默认值校验。
6) 文档与运维  
   - 补充：架构图、注册/打卡/行程时序图；常见故障处理（Redis/MQ/DB）；缓存键/TTL 约定；限流策略与阈值表。  
   - 在 README 或 docs 记录环境启动方式、健康检查、关键指标与告警规则。

## 七、面试问答（由易到难，含场景题）
1) 问：Waitlist 如何防刷与减轻 DB 压力？  
   答：Redis 限流 `waitlist:rate:{open_id}` 60s>30 拒绝；结果缓存 `waitlist:info:{open_id}` 5 分钟；仅 miss 时查库/创建，并检查 active 容量。  
   参考：`internal/service/user.go::GetWaitListInfo`。

2) 问：用户状态如何驱动前端路由？  
   答：`resolveNextStep` 根据 `status`+`phone_verified` 输出 `waitlist|bind_phone|fill_contacts|home`，在 waitlist/登录接口统一返回，前端无需推导。

3) 问：每日打卡提醒如何保证幂等与频控？  
   答：事务内先查当日同类通知任务，存在则跳过；月度 5 条频控（Redis）；Snowflake 生成任务 ID/code，避免重复；出错回滚事务。  
   参考：`internal/service/checkin.go` 批量创建通知任务段。

4) 问：行程超时提醒的策略？  
   答：创建行程时要求返程时间 ≥ 当前+10min；自动提醒开启且延迟 ≤24h 时构建延迟消息（携带 DelaySeconds）；>24h 交给 scheduler；返回消息由上层发布。  
   参考：`internal/service/journey.go` 创建段。

5) 问：支付宝授权如何避免重复账号？  
   答：按 `alipay_open_id` 查，miss 时按 `phone_hash` 合并旧记录；新用户事务创建钱包与额度；唯一约束在 DB 层保护。  
   参考：`internal/service/auth.go::ExchangeAlipayAuthCode`。

6) 问：Redis 挂掉时 waitlist 会怎样？如何改进？  
   答：限流/缓存失败会直查 DB，风险是被打爆；可加本地退化限流或直接 429/503，并记录告警，恢复后再放行；同时对缓存读错不写回。

7) 问：通知/扣费幂等如何保证？  
   答：依赖 `notification_tasks` 状态与 `quota_transactions` 记录；应在消费侧按任务 ID 幂等处理，预扣/确认/退款分阶段；必要时加幂等键或乐观锁，防止重复扣减。

8) 问：如何支持多供应商短信的降级？  
   答：抽象短信发送接口，记录渠道；消费端失败后切换备份渠道，额度流水记录渠道字段；指标分渠道上报，便于熔断/降级。

9) 问：大促场景下，如何保护打卡/行程创建？  
   答：前置 IP/用户限流；热点参数缓存；队列削峰；对打卡任务创建做批量/异步；对行程 creation 加幂等/唯一约束，避免并发爆表。

10) 问：如何处理同一用户重复创建进行中行程？  
    答：目前未限制；改进：对 user+status(ongoing) 加唯一索引或创建前检查；若存在未完成行程直接返回现有行程并拒绝新建。

11) 问：如何验证与调试延迟消息链路？  
    答：在非生产环境设置短延迟，观察 MQ 消费日志与任务状态；配合 otel trace；确保消息体包含幂等 ID（task_id/message_id），消费端先查任务状态再执行扣费/通知。

12) 问：数据库层面的性能瓶颈与优化？  
    答：核心查询已设索引/唯一约束；需关注 `Count`（waitlist 容量）可缓存；通知任务查询日期范围需索引（已存在）；可增加覆盖索引/分区视业务量演进。

## 八、答题提示（面试策略）
- 先描述“入口 → 校验/限流 → 缓存 → DB/MQ → 幂等/事务 → 返回/缓存”标准路径。
- 遇到“故障/扩容”类追问，优先提“降级、熔断、限流、幂等、告警、观测”六件套。
- 对安全/一致性问题，强调唯一约束、幂等键、事务边界、缓存失效策略。
